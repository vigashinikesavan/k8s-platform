# Backup and Restore Notes

## Scope

This setup is a small two-node k3s cluster for a take-home assignment. A full production backup system was not implemented, but the main backup and restore considerations are documented here.

## What should be backed up

Important items:

* Git repository containing Kubernetes manifests, Helm values, scripts, and documentation
* k3s cluster state from the control-plane node
* Application configuration
* cert-manager issuer manifests
* Monitoring configuration

Generated secrets such as TLS certificates and Grafana passwords are not committed to Git.

## k3s state

The control-plane node stores k3s state under:

```text
/var/lib/rancher/k3s/
```

For a single-node k3s server using the default embedded datastore, a backup should include the k3s server database snapshot or relevant datastore files.

In a production setup, scheduled k3s snapshots should be enabled and copied to external storage.

## Restore approach

A basic restore approach would be:

1. Provision replacement VM infrastructure.
2. Install k3s with the same version or a compatible version.
3. Restore the k3s datastore snapshot.
4. Re-apply Kubernetes manifests from Git if needed.
5. Verify nodes, workloads, ingress, TLS, and monitoring.

## Manifests

Most platform configuration is recoverable from Git. After restoring or recreating the cluster, the main Kubernetes resources can be applied again:

```bash
kubectl apply -f kubernetes/cert-manager/
kubectl apply -f kubernetes/sample-app/
./scripts/install-monitoring.sh
```

The monitoring script installs `kube-prometheus-stack` using the Helm values stored in `kubernetes/monitoring/values.yaml`.

## TLS

TLS certificates are generated by cert-manager and stored as Kubernetes Secrets. These generated Secrets are not committed to Git.

If the cluster is rebuilt, cert-manager can request new certificates from Let's Encrypt after the Ingress and ClusterIssuer resources are restored.

## Monitoring data

Prometheus data is useful for troubleshooting but is not treated as critical application data in this assignment. In production, Prometheus retention, persistent storage, and remote write should be considered.

## Improvements for production

* Enable scheduled k3s datastore snapshots
* Store backups outside the cluster
* Test restore regularly
* Use persistent storage for monitoring data
* Document recovery time and recovery point objectives
